Convenient — especially when it's a problem you've seen before.

But picking the chart type is only step one.

Validate the heuristic — don't dismiss it. It genuinely works for common cases. But set up the next beat: the chart type decision is just the top of a long chain of smaller decisions.

then what?

1 chart type — bar, line, scatter, …
2 which variant? stacked vs grouped? shared vs independent axes?
3 colors — which dimension to encode? how many?
4 labels — which values to annotate? where?
5 …
Each of these is a real decision with real tradeoffs. The chart-type heuristic gets you to step 1. Everything below it is still up to you. Decision diagrams (like the "chart chooser" posters) exist but they're clunky — they only go one or two levels deep and they stop exactly where it gets interesting.

can't we just ask a model?

Models are getting good at step 1. Sometimes step 2.

But the micro-decisions — which color encodes which dimension, which values deserve a label, whether a stacked or small-multiples layout better supports your reader's query — these require knowing what you're trying to say.

A model that can't articulate the tradeoff can't reliably make the call. Neither can a chart-chooser diagram.

Don't dismiss AI tools — they're genuinely useful. But they're operating on the same vague heuristics. The missing ingredient is a precise vocabulary for talking about visual structure. That's what sets up GoFish.

  1. The first weight is suspended from the left end of a rope over Pulley A.
  2. The right end of this rope is attached to, and partially supports, the second weight.
  3. Pulley A is suspended from the left end of a rope that runs over Pulley B, and under Pulley C.
  4. Pulley B is suspended from the ceiling.
  5. The right end of the rope that runs under Pulley C is attached to the ceiling.
  6. Pulley C is attached to the second weight, supporting it jointly with the right end of the first rope.

  • [S2→S3] PA suspended over PB under PC
  • What is PC attached to?
  • [S3→S4] irrelevant
  • [S4→S5] PC supported by ceiling
  • What is PA attached to again?

Same problem. Now the spatial structure makes the reasoning easy — follow the ropes with your eyes. Larkin & Simon (1987): diagrammatic representations support "efficient lookups keyed to one element of a relation." This is the whole point. A visualization is a spatial encoding of relational structure that makes certain inferences perceptual rather than logical.

diagrams can efficiently encode relational information in space

Sentential representation requires linear search to discover facts.
"A points to B", "B points to C", "A points to C" — what does A point to?
Diagrammatic representation supports efficient lookups keyed to one element of a relation.
"Everything that points to A" or "everything A points to" are simple perceptual tasks of following lines.

Larkin and Simon. "Why a Diagram is (Sometimes) Worth Ten Thousand Words." 1987

One more beat before the GoFish intro. This is why the structure of a visualization matters so much — it's not decoration, it's computation offloaded to the perceptual system. ~5 min mark.

thinking about graphics structurally, not taxonomically

Instead of asking "which chart type fits my data?" —
ask "what structure do I need to support the queries I want?"

Picking a chart type is a convenient shortcut when the answer is obvious. But structure is the underlying thing.

The pivot. This is where we move from motivation to solution. The chart-type vocabulary is a set of named structures. The interesting question is how to reason about structure directly.

The move: stack()

chart(data).flow(
    spread("lake",    dir="x"),

).mark(
    rect(h="count", fill="species")  # ← fill
)

One structural move. New affordances gained (part-to-whole), old ones traded away (cross-category baseline comparison). ~9 min mark.

The move: connect the stacks into a continuous ribbon

Layer([
  chart(data).flow(
    spread("lake", dir="x", spacing=64),
    derive(lambda d: sorted(d, key="count")),
    stack("species", dir="y"),
  ).mark(
    rect(h="count", fill="species").name("bars")  # ← name it
  ),
  chart(select("bars"))           # ← new layer
    .flow(group("species"))
    .mark(area(opacity=0.8)),
])
Same stacking structure — but a continuous axis opens up trend queries: how does each species change across lakes?
Key: we didn't throw away the stacked structure. We added continuity on top of it. The area mark connects the already-placed rect nodes. ~11 min mark.

The ribbon doesn't just add trend queries — it reorganizes which queries are primary and which are secondary, via Gestalt principles.

You have choice — and now you know how to make it

Three visualizations a typology would treat as unrelated chart types.

But they share the same outer structure and differ only in the inner glyph.

Once you see it structurally, the choice becomes deliberate: which inner query do you want to afford?

There are real tradeoffs — familiarity, legibility, precision. But the query-affordance frame tells you how to reason about them.

And as AI generation gets cheaper, the cost of exploring structural variations is dropping fast.
